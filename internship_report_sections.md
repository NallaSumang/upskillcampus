# Section 5: Proposed Design/Model

The Smart Factory Dashboard is architected using a modern, decoupled client-server model leveraging React.js for the frontend and Spring Boot for the backend. The frontend operates as a Single Page Application (SPA), providing a highly responsive and dynamic user interface. React’s functional components and hooks manage the local state and UI rendering, while Axios is utilized to handle asynchronous HTTP requests to the backend API. Tailwind CSS is integrated to ensure rapid, consistent, and responsive styling across all dashboard components.

On the server side, a Spring Boot REST API handles business logic, data persistence, and routing. The application uses Spring Web to expose endpoints and Spring Data JPA to interface seamlessly with the underlying database. The `AssetController` receives HTTP requests from the React frontend, processes them via the `AssetService`, and interacts with the `AssetRepository`. To minimize boilerplate code and enhance readability, Lombok is incorporated into the entity definitions and data models. Cross-Origin Resource Sharing (CORS) is explicitly configured on the controllers to securely allow requests originating from the React development server.

Data persistence is managed by an H2 in-memory database, which provides rapid read/write operations without the overhead of external database dependencies, making it ideal for self-contained, out-of-the-box deployments. The data flows seamlessly: when a user interacts with the React dashboard (e.g., clicking a button to change a machine’s status), an Axios PUT request is sent to the Spring Boot API. The controller receives this request, updates the JPA `Asset` entity, and saves the new state to the H2 database. The updated record is then serialized back into JSON and returned to the frontend, where the React state updates to instantly reflect the new status on the dashboard.

# Section 6: Performance Test

The system was subjected to performance testing to validate its capability to handle concurrent data requests with minimal API latency. A primary constraint for this architecture is the single-node nature of the H2 in-memory database; while highly performant for localized execution, it could theoretically experience lock contention under massive, sustained write loads. The performance test specifically targets the critical `GET /api/assets` endpoint using JUnit 5 and `MockMvc`. The test case simulates a load of 100 concurrent requests, accurately measuring the total and average execution times using Spring’s `StopWatch` utility. The primary assertion parameter requires that the system processes these requests and delivers the JSON payload with an average latency of under 200ms.

**Performance Outcome**: Under the simulated load of 100 sequential requests using `MockMvc`, the system demonstrated exceptional stability and low latency. The Spring Boot backend, coupled with the fast, in-memory H2 database, effectively processed the data retrieval operations well within the acceptable thresholds. The logging output verified that the average response time per request remained significantly below the 200ms limit, proving that the architecture is highly capable of supporting rapid, dashboard-style polling or multiple concurrent users without degrading the user experience.
